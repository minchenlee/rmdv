# rmdv v0.2 — Performance

**Status:** Draft
**Date:** 2026-05-10
**Axis:** Performance (1 of 4 in v0.2–v0.5 differentiation roadmap)

## Context

rmdv shipped v0.1.0 as the second serious entrant in the native cross-platform OSS markdown viewer space. Marky (Tauri+Rust, 254★, mac+Linux) is the only direct peer. rmdv's structural advantages are Windows support and a pure-Rust GUI via Iced (no embedded WebView).

To convert "no WebView" from a talking point into a measurable advantage, v0.2 focuses on performance — the dimension where pure-Rust + Iced should beat Tauri+React most clearly.

Roadmap: v0.2 Performance → v0.3 Reading experience → v0.4 Workflow → v0.5 Distribution.

## Goals

- Cold start under 100 ms (measured, optimized, published)
- Smooth scroll and constant memory on 1 MB / ~10k-line markdown documents
- Hot reload re-highlights only changed code blocks, not the full document

## Non-goals

- Streaming parse for >10 MB files (defer to v0.3+ if demand emerges)
- Off-main-thread parsing or GPU text shaping beyond Iced defaults
- Replacing the Iced rendering pipeline

## Workstreams

Three independent workstreams, shippable separately, combined as v0.2.

### A. Cold start benchmark and optimization

Add a `--benchmark-startup` flag in `main.rs` that records timestamps at four checkpoints:

1. process entry
2. font system load complete
3. window created
4. first frame painted

Print durations to stdout. Run on macOS arm64 release build, take median of 10 runs.

Profile dominant costs. Expected hotspots:

- `font_system().write().load_system_fonts()` at startup — defer until first text render
- Tree-sitter grammar initialization — lazy-load each grammar on first use of that language

Publish numbers in README and `docs/benchmarks.md`. Compare against Marky if reproducible on the same machine.

### B. Virtual scrolling

Refactor the renderer so the parsed AST produces a list of "blocks" (paragraph, heading, code, image, table, list). Each block is one renderable unit with a stable index.

Use Iced's `scrollable` viewport awareness to compute the visible block range. Only build widgets for visible blocks plus ~5 blocks of padding above and below.

Block height estimation:

- First-time estimate from cheap heuristic (line count × font line-height + paddings)
- Real height measured on first paint, cached for subsequent frames
- Cache keyed by block content hash

Maintain stable scroll position when blocks above the viewport change height (anchor to first-visible block ID).

### C. Incremental tree-sitter reparse

Current path: `notify` watcher → re-read file → re-parse markdown → re-highlight all code blocks.

New path: on file change, diff old vs new block list. For each unchanged code block, reuse cached highlight tree. For changed blocks, reparse only that block's source with the relevant grammar.

Cache: `HashMap<(Lang, u64 hash), HighlightedTree>` with LRU eviction (last ~200 blocks).

Tree-sitter's incremental API operates on byte-edits. We get whole-file rewrites from disk. Skip incremental at the AST layer; cache at the code-block layer only — simpler, captures ~90% of the win.

## Architecture

No major file restructure. Touch points:

| File | Change |
|---|---|
| `src/main.rs` | `--benchmark-startup` flag, timing prints |
| `src/app.rs` | Defer font loading; viewport-aware render call |
| `src/render.rs` | Block-list rebuild; visible-range computation; height cache |
| `src/highlight.rs` | Block-keyed LRU cache |
| `src/parser.rs` | Emit block list with stable IDs (line-span hash) |
| `src/watch.rs` | Pass old-AST hint to update path |
| `benches/cold_start.rs` (new) | Criterion bench |
| `benches/render_1mb.rs` (new) | Criterion bench |
| `tests/perf_smoke.rs` (new) | Assert 1 MB doc first frame < 500 ms |

## Testing

- `cargo bench` baselines for cold start and 1 MB render
- Existing `insta` snapshot tests must still pass — virtual scroll must produce the same visual output at any given scroll position
- Manual: scroll a generated 10k-line document, watch memory in Activity Monitor — should stay flat
- Hot reload test: edit one code block in a 100-block document, assert only that block's highlight cache entry is invalidated

## Risks and mitigations

**Iced viewport callbacks.** Iced's `scrollable` doesn't expose viewport-change callbacks in the stable API. Mitigation: prototype virtual scroll first; if Iced can't, fall back to "render N blocks initially, expand on scroll-to-end" — less elegant but ships.

**Block height estimation wrong.** Causes scroll jumpiness on first encounter with a block. Mitigation: measure-then-cache; accept first-scroll wobble as acceptable.

**Tree-sitter whole-file edits.** Incremental API expects known byte-ranges. Mitigation: skip incremental at AST layer entirely; cache at the code-block layer.

## Success criteria

- Cold start median < 100 ms on macOS arm64 release build
- 1 MB / 10k-line document scrolls smoothly (60fps) with flat memory
- Hot reload of a single code block in a 100-block doc completes in < 50 ms
- README and `docs/benchmarks.md` publish reproducible numbers
